# Database Migration Guide

**ORM:** Prisma
**Database:** PostgreSQL 15
**Rule:** All schema changes must go through Prisma migrations. Never modify the database schema directly in any environment.

---

## Migration Workflow

### Development

```bash
# 1. Edit packages/db/prisma/schema.prisma
# 2. Generate and apply the migration locally
cd packages/db
pnpm db:migrate          # prisma migrate dev â€” creates migration file + applies it

# 3. Regenerate the Prisma client
pnpm db:generate
```

### Production (automated in CD pipeline)

The `deploy.yml` CD pipeline runs `prisma migrate deploy` via a one-off ECS task **before** the new app version starts. This ensures:

1. RDS snapshot is taken (rollback point).
2. Migrations are applied to the live database.
3. New app version starts (it already understands the new schema).

### Migration naming convention

Prisma generates the timestamp prefix automatically. Add a descriptive snake_case suffix:

```
20260101120000_add_assay_discrepancy_flag_to_transactions
20260201090000_rename_client_mobile_to_mobile_phone
20260301000000_add_client_deletion_requested_at
```

### Review checklist before merging a migration

- [ ] Does the migration run in under 30 seconds on a representative dataset? (Use `EXPLAIN ANALYZE` on the migration SQL.)
- [ ] Is it safe to run while the old app version is still serving traffic? (see patterns below)
- [ ] Does it have a rollback path? (document in PR if not)
- [ ] Has it been tested against a copy of the production dataset?

## Zero-Downtime Migration Patterns

### Adding a nullable column (safe)

The simplest case â€” PostgreSQL adds the column without scanning or locking existing rows.

```sql
-- Generated by Prisma for: field String?
ALTER TABLE "transactions" ADD COLUMN "assayDiscrepancyFlag" BOOLEAN;
```

**Prisma schema:**

```prisma
assayDiscrepancyFlag  Boolean?
```

**Deployment order:** Migrate first, then deploy new app. The old app ignores the new column; the new app writes to it.

---

### Adding a non-nullable column (requires default or two-phase migration)

Without a `DEFAULT`, PostgreSQL must update every existing row â€” this takes a table lock on large tables.

**Option A â€” add with a `DEFAULT` (safe for most columns):**

```sql
-- Safe: Postgres 11+ stores the default in the catalog and does not rewrite the table
ALTER TABLE "transactions" ADD COLUMN "version" INTEGER NOT NULL DEFAULT 1;
```

```prisma
version  Int  @default(1)
```

**Option B â€” two-phase migration (for large tables where even a brief metadata lock is unacceptable):**

Phase 1 (ship with old app): Add as nullable.

```sql
ALTER TABLE "transactions" ADD COLUMN "version" INTEGER;
```

Phase 2 (after old app is fully drained): Backfill + add NOT NULL constraint.

```sql
-- Backfill in batches to avoid long transactions
UPDATE "transactions" SET "version" = 1 WHERE "version" IS NULL;
ALTER TABLE "transactions" ALTER COLUMN "version" SET NOT NULL;
ALTER TABLE "transactions" ALTER COLUMN "version" SET DEFAULT 1;
```

---

### Renaming a column (two-phase â€” never rename directly in production)

A direct rename locks the table and breaks any in-flight queries using the old name.

**Phase 1 â€” add the new column, dual-write:**

```sql
ALTER TABLE "clients" ADD COLUMN "mobilePhone" TEXT;
-- Backfill
UPDATE "clients" SET "mobilePhone" = "mobile" WHERE "mobilePhone" IS NULL;
```

Deploy app version that writes to **both** `mobile` and `mobilePhone`.

**Phase 2 â€” remove the old column (after Phase 1 is fully deployed):**

```sql
ALTER TABLE "clients" DROP COLUMN "mobile";
```

Deploy app version that only uses `mobilePhone`.

> **Important:** Prisma migrations do not support two-phase column renames natively. Split this across two separate migration files in two separate deployments.

---

### Adding an index on a large table (use `CONCURRENTLY`)

`CREATE INDEX` without `CONCURRENTLY` takes a `ShareLock` which blocks writes for the duration.

**Step 1 â€” write the migration SQL manually (Prisma does not generate `CONCURRENTLY`):**

```sql
-- Migration file: 20260301000000_add_transactions_created_at_idx/migration.sql
-- This migration must be run manually; do NOT run via prisma migrate deploy
-- (CONCURRENTLY cannot run inside a transaction)

CREATE INDEX CONCURRENTLY IF NOT EXISTS "transactions_createdAt_idx"
  ON "transactions" ("createdAt" DESC);
```

**Step 2 â€” mark as applied without running it through Prisma:**

```bash
# After running the SQL manually:
DATABASE_URL=<prod_url> npx prisma migrate resolve --applied 20260301000000_add_transactions_created_at_idx
```

**Step 3 â€” add the `@@index` to schema.prisma** so Prisma's introspection stays consistent:

```prisma
@@index([createdAt(sort: Desc)])
```

> Tip: `CREATE INDEX CONCURRENTLY` can take minutes on a large table. Monitor `pg_stat_progress_create_index` during the operation.

---

### Dropping a column (two-phase)

The old app will error if it tries to read a column that no longer exists.

**Phase 1 â€” stop selecting the column in the app code** (deploy first).

**Phase 2 â€” drop the column:**

```sql
ALTER TABLE "clients" DROP COLUMN "legacyField";
```

---

### Changing a column type (high risk â€” always two-phase)

Type changes often rewrite the table. Plan carefully:

1. Add the new column with the target type.
2. Backfill with a cast.
3. Deploy app that dual-reads (old and new).
4. Remove old column in a follow-up migration.

## Seed Data

The seed file at `packages/db/prisma/seed.ts` is **idempotent** â€” it uses `upsert` throughout and is safe to re-run in any environment:

```bash
cd packages/db
pnpm db:seed
```

Seed creates: default SUPER_ADMIN user, sample agent, Rand Refinery, 3 LME price records, system settings defaults.
